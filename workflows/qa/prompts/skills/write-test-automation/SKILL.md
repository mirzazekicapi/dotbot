# Write Test Automation

Guide for generating test automation code from detailed test case documents. The automation code is written in an external test repository, using the framework configured in `qa.test_framework` settings.

## Prerequisites

All of these must be in place:

1. **Test case files** — `.bot/workspace/product/test-cases/*.md` must exist (generated by workflow 05)
2. **QA settings configured** — `qa.test_repo_path` must point to a valid, locally cloned git repository
3. **Framework selected** — `qa.test_framework` must be set to a valid preset key

If any prerequisite is missing, stop and surface the gap to the operator.

## Inputs to Collect

```
.bot/workspace/product/test-cases/*.md        # test case documents (required)
.bot/workspace/product/test-plan.md           # for traceability context
.bot/workspace/product/tech-stack.md          # for environment details
qa.test_repo_path (settings)                  # external test repo location
qa.test_framework (settings)                  # framework preset key
qa.test_framework_presets (settings)          # framework configuration
```

Before generating code, also scan the external test repo to understand:
- Existing directory structure and naming conventions
- Existing page objects, helpers, fixtures, or utilities to reuse
- Test runner configuration files (playwright.config.ts, pom.xml, etc.)
- Any existing test patterns to follow for consistency

## Framework Resolution

1. Read `qa.test_framework` from settings (e.g., `"playwright-ts"`)
2. Look up the preset in `qa.test_framework_presets`:
   - `language` — programming language for code generation
   - `test_runner` — command to execute tests
   - `file_pattern` — naming convention for test files
   - `base_dir` — root directory for tests within the test repo
3. If preset is `"custom"`, read custom values from settings

## Code Generation Patterns

### Universal Patterns (All Frameworks)

**Test function naming:**
```
test_TC_I_01_descriptive_name     # maps to TC-I-01
test_TC_E_01_descriptive_name     # maps to TC-E-01
test_TC_UAT_01_descriptive_name   # maps to TC-UAT-01 (if automated)
```

**Traceability comments:**
Every test file and test function must include a comment tracing back to the source:
```
// Source: .bot/workspace/product/test-cases/{group-id}-{slug}.md
// Test Case: TC-I-01 — [test case title]
// Scenario: I-01 — [scenario name from test plan]
```

**File organisation:**
```
{base_dir}/
  {group-slug}/
    {group-slug}.{file_pattern}     # test file per group
  pages/                            # page objects (UI tests)
  fixtures/                         # test data factories
  helpers/                          # shared utilities
```

**Test data:**
- Extract test data from the test case "Test Data" table
- Create fixtures or factories for reusable test data
- Never hardcode credentials — use environment variables
- Use realistic but synthetic data (no real PII)

### UI Test Patterns (Playwright, Selenium, Cypress)

**Page Object Model:**
- One page object per significant page or component
- Page objects encapsulate selectors and actions
- Tests call page object methods, not raw selectors
- Page objects return other page objects for navigation flows

**Selector strategy:**
- Prefer `data-testid` attributes
- Fall back to accessible roles and labels (`getByRole`, `getByLabel`)
- Avoid CSS class selectors (brittle)
- Avoid XPath unless absolutely necessary

**Wait strategy:**
- Use explicit waits for dynamic content
- Wait for specific conditions (element visible, API response), not fixed timeouts
- Handle loading states and animations

### API Test Patterns (Supertest, REST Assured)

**API client abstraction:**
- One client class per API resource or domain area
- Methods map to API operations (create, get, update, delete)
- Base URL from environment configuration
- Auth token handling in shared setup

**Request/response validation:**
- Validate status codes explicitly
- Validate response body structure (schema validation where possible)
- Validate headers when relevant (content-type, cache-control)
- Store response data for subsequent steps in multi-step flows

**Test isolation:**
- Each test creates its own test data (no shared state between tests)
- Clean up created resources in teardown
- Use unique identifiers to avoid collisions in parallel runs

## Per-Framework Guidance

### Playwright (TypeScript)

```typescript
// tests/{group-slug}/{group-slug}.spec.ts
import { test, expect } from '@playwright/test';
import { LoginPage } from '../pages/login-page';

// Source: .bot/workspace/product/test-cases/{group-id}-{slug}.md
// Test Case: TC-E-01 — User login with valid credentials
// Scenario: E-01 — Successful authentication flow
test('TC-E-01: user login with valid credentials', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login(process.env.TEST_USER!, process.env.TEST_PASS!);
  await expect(page.getByRole('heading', { name: 'Dashboard' })).toBeVisible();
});
```

- Use `test.describe` blocks per scenario group
- Use `test.beforeEach` for common setup (navigation, auth)
- Use fixtures for reusable test data and page objects
- Configure `playwright.config.ts` for parallel execution and retries

### Selenium (C#)

```csharp
// Tests/{GroupSlug}/{GroupSlug}Tests.cs
using NUnit.Framework;

// Source: .bot/workspace/product/test-cases/{group-id}-{slug}.md
[TestFixture]
public class GroupSlugTests : BaseTest
{
    // Test Case: TC-E-01 — User login with valid credentials
    // Scenario: E-01 — Successful authentication flow
    [Test]
    public void TC_E_01_UserLoginWithValidCredentials()
    {
        var loginPage = new LoginPage(Driver);
        loginPage.NavigateTo();
        loginPage.Login(TestConfig.TestUser, TestConfig.TestPassword);
        Assert.That(Driver.FindElement(By.CssSelector("h1")).Text, Is.EqualTo("Dashboard"));
    }
}
```

- Use `[SetUp]` and `[TearDown]` for browser lifecycle
- Use `BaseTest` class for shared configuration
- Use `[Category("smoke")]` or `[Category("regression")]` from test case tags
- Use `[TestCase]` for data-driven tests from test data tables

### Selenium (Java)

```java
// src/test/java/com/project/tests/GroupSlugTest.java
import org.junit.jupiter.api.*;

// Source: .bot/workspace/product/test-cases/{group-id}-{slug}.md
class GroupSlugTest extends BaseTest {

    // Test Case: TC-E-01 — User login with valid credentials
    // Scenario: E-01 — Successful authentication flow
    @Test
    @Tag("smoke")
    void TC_E_01_userLoginWithValidCredentials() {
        LoginPage loginPage = new LoginPage(driver);
        loginPage.navigateTo();
        loginPage.login(testConfig.getTestUser(), testConfig.getTestPassword());
        assertEquals("Dashboard", driver.findElement(By.cssSelector("h1")).getText());
    }
}
```

- Use JUnit 5 `@Tag` annotations from test case tags
- Use `@BeforeEach` / `@AfterEach` for setup/teardown
- Use `@ParameterizedTest` for data-driven tests

### Cypress (TypeScript)

```typescript
// cypress/e2e/{group-slug}/{group-slug}.cy.ts

// Source: .bot/workspace/product/test-cases/{group-id}-{slug}.md
describe('[Group Name]', () => {
  // Test Case: TC-E-01 — User login with valid credentials
  // Scenario: E-01 — Successful authentication flow
  it('TC-E-01: user login with valid credentials', () => {
    cy.visit('/login');
    cy.get('[data-testid="email"]').type(Cypress.env('TEST_USER'));
    cy.get('[data-testid="password"]').type(Cypress.env('TEST_PASS'));
    cy.get('[data-testid="submit"]').click();
    cy.get('h1').should('contain', 'Dashboard');
  });
});
```

- Use `cy.intercept` for API mocking when needed
- Use custom commands for reusable actions
- Use fixtures for test data (`cy.fixture`)

### API — Supertest (TypeScript)

```typescript
// tests/api/{group-slug}/{group-slug}.test.ts
import request from 'supertest';

// Source: .bot/workspace/product/test-cases/{group-id}-{slug}.md
describe('[Group Name] API', () => {
  // Test Case: TC-I-01 — Create resource via POST
  // Scenario: I-01 — Resource creation flow
  it('TC-I-01: should create resource via POST', async () => {
    const res = await request(baseUrl)
      .post('/api/resources')
      .set('Authorization', `Bearer ${authToken}`)
      .send({ name: 'Test Resource' })
      .expect(201);

    expect(res.body).toHaveProperty('id');
    expect(res.body.name).toBe('Test Resource');
  });
});
```

### API — REST Assured (Java)

```java
// src/test/java/com/project/api/GroupSlugApiTest.java
import static io.restassured.RestAssured.*;

// Source: .bot/workspace/product/test-cases/{group-id}-{slug}.md
class GroupSlugApiTest extends BaseApiTest {

    // Test Case: TC-I-01 — Create resource via POST
    // Scenario: I-01 — Resource creation flow
    @Test
    void TC_I_01_createResourceViaPost() {
        given()
            .header("Authorization", "Bearer " + authToken)
            .contentType(ContentType.JSON)
            .body(new ResourceRequest("Test Resource"))
        .when()
            .post("/api/resources")
        .then()
            .statusCode(201)
            .body("name", equalTo("Test Resource"));
    }
}
```

### Appium (Java)

```java
// src/test/java/com/project/mobile/GroupSlugMobileTest.java
import io.appium.java_client.*;

// Source: .bot/workspace/product/test-cases/{group-id}-{slug}.md
class GroupSlugMobileTest extends BaseMobileTest {

    // Test Case: TC-E-01 — User login on mobile
    // Scenario: E-01 — Mobile authentication flow
    @Test
    void TC_E_01_userLoginOnMobile() {
        LoginScreen loginScreen = new LoginScreen(driver);
        loginScreen.enterEmail(testConfig.getTestUser());
        loginScreen.enterPassword(testConfig.getTestPassword());
        loginScreen.tapLogin();
        assertTrue(new DashboardScreen(driver).isDisplayed());
    }
}
```

- Use screen objects (mobile equivalent of page objects)
- Handle platform differences (iOS vs Android) in screen objects
- Use `@Tag("android")` / `@Tag("ios")` for platform-specific tests

## Anti-Patterns

- **Do not generate tests without reading the test repo first** — existing patterns must be followed for consistency
- **Do not hardcode test data** — use fixtures, factories, or environment variables
- **Do not skip traceability comments** — every test must trace back to a test case ID and scenario ID
- **Do not mix test levels** — a UI test should not make direct database assertions; an API test should not drive a browser
- **Do not generate UAT test cases as automation** — UAT scenarios are manual by definition unless explicitly tagged as automatable
- **Do not ignore existing helpers** — reuse page objects, API clients, and utilities already in the test repo
- **Do not use `sleep` or fixed waits** — use explicit wait conditions

## Output Checklist

- [ ] Every automatable test case has a corresponding test function
- [ ] Test function names include the test case ID (TC-xx-xx)
- [ ] Traceability comments reference source test case file and scenario ID
- [ ] Page objects or API clients are created for reusable interactions
- [ ] Test data uses fixtures/factories, not hardcoded values
- [ ] No credentials in source code (environment variables only)
- [ ] Tests follow the existing repo's naming and structure conventions
- [ ] Tests can be run with the configured `test_runner` command
- [ ] No real PII in test data
