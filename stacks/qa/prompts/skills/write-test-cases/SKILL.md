# Write Test Cases

Guide for producing detailed, executable test case documents from the high-level test plan. Each test plan scenario (I-xx, E-xx, UAT-xx) is expanded into one or more granular test cases with step-by-step procedures.

## Prerequisites

Both of these must exist before writing test cases:

1. **Test plan** — `.bot/workspace/product/test-plan.md` must exist (generated by workflow 04)
2. **Product specifications** — at least one of: `mission.md`, `entity-model.md`, PRD, change request

If either is missing, stop and surface the gap to the operator. Test cases without a test plan have no traceability.

## Inputs to Collect

```
.bot/workspace/product/test-plan.md           # source scenarios (required)
.bot/workspace/product/mission.md             # core goals and principles
.bot/workspace/product/entity-model.md        # data model and relationships
.bot/workspace/product/tech-stack.md          # runtime, frameworks, E2E tooling
.bot/workspace/product/task-groups.json       # group-level scope
task_list (MCP)                               # all tasks with acceptance criteria
```

Read all inputs before generating test cases. The test plan is the primary source; product docs provide context for realistic test data and edge cases.

## Output Structure

Output directory: `.bot/workspace/product/test-cases/`

One file per task group: `{group-id}-{group-slug}.md`

### File Format

```markdown

**Generated:** YYYY-MM-DD

| P2       | N           | N           | N     |

| P4       | N           | N           | N     |

**Scenario:** I-01 — [scenario name from test plan]
**Type:** Integration
```

## Test Case ID Convention

Test case IDs are derived from scenario IDs with a `TC-` prefix:

| Scenario ID | Test Case IDs |
|-------------|---------------|
| I-01        | TC-I-01, TC-I-01a, TC-I-01b (if multiple cases per scenario) |
| E-01        | TC-E-01, TC-E-01a, TC-E-01b |
| UAT-01      | TC-UAT-01, TC-UAT-01a, TC-UAT-01b |

Use letter suffixes (a, b, c) when a single scenario requires multiple test cases (e.g., happy path + error path + edge case).

## Derivation Rules

### From integration scenarios (I-xx)

- Focus on **API contracts, data flow, and component boundaries**
- Steps should describe API calls, request payloads, and expected response structures
- Include database state verification where applicable
- Specify what is real vs. stubbed (e.g., "real DB, mocked email service")
- Always automatable unless they require manual infrastructure setup

### From E2E / acceptance scenarios (E-xx)

- Focus on **complete user journeys from UI to persistence**
- Steps should describe exact UI interactions (click, type, navigate, wait)
- Include visual verification points (element visible, text matches, state changes)
- Specify browser or device requirements if applicable
- Usually automatable with UI test frameworks (Playwright, Selenium, Cypress)

### From UAT scenarios (UAT-xx)

- Focus on **business outcomes in plain language**
- Steps should be understandable by non-technical stakeholders
- No code references, API endpoints, or technical jargon
- Include business context ("as a manager, verify that...")
- Always marked as manual-only (UAT requires human judgement)

## Priority Classification

| Priority | Criteria | When to assign |
|----------|----------|----------------|
| P1 — Critical | Core functionality, security, data integrity | Feature cannot ship without this passing |
| P2 — High | Important workflows, key user journeys | Major user impact if broken |
| P3 — Medium | Secondary features, edge cases, error handling | Noticeable but workaround exists |
| P4 — Low | Cosmetic, nice-to-have, rare edge cases | Minimal user impact |

Derive priority from:
- Risk areas in the test plan (high risk = P1/P2)
- Acceptance criteria importance (must-have = P1, should-have = P2/P3)
- User-facing vs. internal (user-facing = higher priority)

## Tagging Convention

Apply one or more tags to each test case:

| Tag | When to use |
|-----|-------------|
| `api` | Tests API endpoints or contracts |
| `database` | Tests data persistence or queries |
| `auth` | Tests authentication or authorisation |
| `ui` | Tests user interface interactions |
| `e2e` | Tests full user journey end-to-end |
| `manual` | Requires human judgement (UAT, exploratory) |
| `regression` | Should be re-run after any change to the area |
| `smoke` | Quick check that core functionality works |

## Anti-Patterns

- **Do not copy scenario descriptions verbatim** — expand them into concrete, step-by-step procedures
- **Do not use vague steps** ("verify it works") — every step must have an observable, verifiable expected result
- **Do not skip preconditions** — test cases must be independently executable
- **Do not mix test types** — an integration test case should not include UI steps; an E2E test case should not verify internal database state directly
- **Do not invent scenarios** — every test case must trace back to a scenario ID in the test plan
- **Do not use real PII** — use realistic but synthetic test data

## Output Checklist

- [ ] Every scenario ID from test-plan.md has at least one test case
- [ ] Every test case has a unique TC-xx ID
- [ ] Every test case has preconditions, steps with expected results, and pass criteria
- [ ] Every test case is tagged (type, priority, automatable, tags)
- [ ] UAT test cases use business language only (no technical terms)
- [ ] Integration test cases specify real vs. stubbed components
- [ ] E2E test cases describe exact UI interactions
- [ ] Test data is realistic but synthetic (no real PII)
- [ ] Summary table at top of each file is accurate
- [ ] No absolute local paths in the document
